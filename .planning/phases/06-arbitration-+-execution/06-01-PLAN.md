    - submitIntent with priority 4 then priority 2 for same target -> resolve returns dv intent (lower number wins)
    - submitIntent with priority 2 then priority 4 for same target -> resolve still returns dv (does not overwrite with lower priority)
    - Different targets from different sources -> both stored independently
    - clearSource('optimizer') removes all optimizer intents, leaves others
    - resolveAll() returns Map of all current winners
    - resolve(target) returns null for unknown target
    - getOverridden() returns list of intents that were submitted but lost to higher-priority

  <action>
    Create `dvhub/core/arbitrator.js` exporting `createArbitrator({ log } = {})`.

    Factory returns: `{ submitIntent, resolve, resolveAll, clearSource, getOverridden, clear }`.

    Priority map (constants, not configurable):
    ```
    const PRIORITY_MAP = { system: 1, dv: 2, manual: 3, optimizer: 4 };
    ```

    Internal state: `activeIntents = new Map()` keyed by target string. Each value: `{ source, priority, value, timestamp, reason }`.

    `submitIntent(intent)`:
    - Validate intent has source, priority, targets, timestamp
    - For each `[target, value]` in `Object.entries(intent.targets)`:
      - Get current winner for target
      - If no current OR `intent.priority <= current.priority`: replace (lower number = higher priority, equal priority replaces for freshness)
      - If intent.priority > current.priority: record as overridden (store in overridden array with { source, priority, target, value, overriddenBy: current.source })
    - Return `{ applied: [...targets that won], overridden: [...targets that lost] }`

    `resolve(target)`: return `activeIntents.get(target) || null`

    `resolveAll()`: return `new Map(activeIntents)`

    `clearSource(source)`: delete all entries where entry.source === source. Return count of cleared entries.

    `getOverridden()`: return copy of overridden array (last N entries, cap at 100).

    `clear()`: clear activeIntents Map and overridden array.

    Use `log?.info()` and `log?.warn()` for optional structured logging (Pino style). Do NOT require log.

    Create `dvhub/test/arbitrator.test.js` using node:test with `describe/it` pattern matching existing tests (see dvhub/test/event-bus.test.js for style). Test all behaviors listed above.

  <behavior>
    - executeCommand logs command:sent to db.insertControlEvent before HAL write
    - executeCommand calls hal.writeControl(target, value)
    - executeCommand reads back via hal.readMeter() after write
    - When readback value matches commanded value within threshold -> logs command:verified
    - When readback deviates beyond threshold -> logs command:deviation AND emits 'exec:deviation' event on eventBus
    - When readback field is unavailable for a target -> logs readback:unavailable, no deviation alert
    - Configurable thresholds per target type: numeric targets (gridSetpointW) use absolute W threshold, boolean targets (feedExcessDcPv) use exact match
    - executeCommand returns { success, target, value, readback, deviation }
  </behavior>
  <action>
    Create `dvhub/core/executor.js` exporting `createExecutor({ hal, db, eventBus, log, config } = {})`.

    Config defaults (merged with provided config):
    ```javascript
    const DEFAULT_CONFIG = {
      readbackEnabled: true,
      readbackDelayMs: 50,         // ms to wait before readback (register settle time)
      thresholds: {
        gridSetpointW: 500,        // W absolute deviation tolerance
        chargeCurrentA: 2,         // A absolute deviation tolerance
        minSocPct: 5,              // % absolute deviation tolerance
        // Boolean DV targets have no threshold -- exact match or unavailable
      }
    };
    ```

    Readback mapping (target -> meter field):
    ```javascript
    const READBACK_MAP = {
      gridSetpointW: 'gridPower',
      // chargeCurrentA: no direct readback available
      // minSocPct: 'soc' (but SoC changes slowly -- wide tolerance)
      // feedExcessDcPv: no readback
      // dontFeedExcessAcPv: no readback
    };
    ```

    `async executeCommand(command)`:
    1. Destructure `{ source, priority, target, value, reason }` from command
    2. Create timestamp `new Date()`
    3. Log command:sent via `db.insertControlEvent({ ts, type: 'command:sent', source, severity: 'info', message: \`${source} -> ${target} = ${value}\`, details: { priority, target, value, reason } })`
    4. Call `const result = await hal.writeControl(target, value)`
    5. If `readbackEnabled` and `READBACK_MAP[target]`:
       a. Wait `readbackDelayMs` via `await new Promise(r => setTimeout(r, readbackDelayMs))`
       b. Call `const meter = await hal.readMeter()`
       c. Extract readback value: `meter[READBACK_MAP[target]]`
       d. Compute deviation: `Math.abs(readback - value)`
       e. Get threshold: `config.thresholds[target]`
       f. If deviation > threshold:
          - Log command:deviation via db.insertControlEvent with severity 'warn'
          - Emit `eventBus.emit({ type: 'exec:deviation', target, commanded: value, readback, deviation, threshold, source, timestamp: Date.now() })`
          - Return `{ success: true, target, value, readback, deviation, deviationAlert: true }`
       g. Else: Log command:verified via db.insertControlEvent, return `{ success: true, target, value, readback, deviation, deviationAlert: false }`
    6. If readback unavailable: Log with message 'readback:unavailable' and return `{ success: true, target, value, readback: null, deviation: null, deviationAlert: false }`

    `getConfig()`: return current merged config (for API status endpoint).

    Create `dvhub/test/executor.test.js` using node:test. Use mock objects for hal, db, and eventBus:
    - `mockHal`: `writeControl` resolves `{ success: true }`, `readMeter` resolves configurable meter object
    - `mockDb`: `insertControlEvent` records calls in array for assertion
    - `mockEventBus`: `emit` records calls in array for assertion

    Test all behaviors listed above. Use `readbackDelayMs: 0` in tests for speed.
