# Phase 7: Deployment - Research

**Researched:** 2026-03-14
**Domain:** Docker Compose orchestration, systemd, hybrid deployment, compose-manager
**Confidence:** HIGH

## Summary

DVhub Phase 7 requires three deployment modes: native-only (systemd, no Docker), hybrid (DVhub native + optimizer containers), and full-Docker (everything containerized). The existing install.sh already handles native deployment with systemd, Node.js installation, and legacy migration. The optimizer adapters (EOS, EMHASS) and EVCC bridge already exist with hardcoded default URLs matching the standard container ports (EOS: 8503, EMHASS: 5000, EVCC: 7070).

All three external services (EOS, EMHASS, EVCC) publish multi-arch Docker images supporting both amd64 and arm64. Docker Compose profiles provide an elegant mechanism to switch between deployment modes from a single compose.yaml. The compose-manager should use raw `child_process.execFile` calling the `docker compose` CLI rather than adding npm dependencies -- this aligns with the project's minimal-dependency philosophy and avoids the maintenance risk of dockerode-compose.

**Primary recommendation:** Use Docker Compose profiles (`--profile hybrid`, `--profile full`) with `deploy.resources.limits` for CPU/memory caps, a compose-manager module using `child_process.execFile('docker', ['compose', ...])`, and enhance install.sh to use `npm ci` with the existing lockfile.

### Pattern 1: Docker Compose Profiles for Deployment Modes
**What:** A single `docker-compose.yaml` with three profile groups
**When to use:** Always -- this is the deployment orchestration file

```yaml
# Source: Docker Compose Profiles documentation
services:
  # --- Optimizer services (hybrid + full profiles) ---
  eos:
    image: akkudoktor/eos:0.2.0
    profiles: ["hybrid", "full"]
    ports:
      - "8503:8503"
      - "8504:8504"
    environment:
      OPENBLAS_NUM_THREADS: "1"
      OMP_NUM_THREADS: "1"
      MKL_NUM_THREADS: "1"
      EOS_SERVER__HOST: "0.0.0.0"
      EOS_SERVER__PORT: "8503"
      EOS_SERVER__EOSDASH_HOST: "0.0.0.0"
      EOS_SERVER__EOSDASH_PORT: "8504"
    volumes:
      - eos-data:/data
    deploy:
      resources:
        limits:
          cpus: '1.0'
          memory: 512M
        reservations:
          cpus: '0.25'
          memory: 256M
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8503/v1/health"]
      interval: 30s
      timeout: 5s
      retries: 3

  emhass:
    image: ghcr.io/davidusb-geek/emhass:v0.17.0
    profiles: ["hybrid", "full"]
    ports:
      - "5000:5000"
    volumes:
      - emhass-data:/share
    deploy:
      resources:
        limits:
          cpus: '1.0'
          memory: 512M
        reservations:
          cpus: '0.25'
          memory: 256M
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:5000/"]
      interval: 30s
      timeout: 5s
      retries: 3

  evcc:
    image: evcc/evcc:0.300.3
    profiles: ["hybrid", "full"]
    ports:
      - "7070:7070"
      - "8887:8887"
    volumes:
      - evcc-data:/root/.evcc
    deploy:
      resources:
        limits:
          cpus: '0.5'
          memory: 256M
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "wget", "--spider", "-q", "http://localhost:7070/"]
      interval: 30s
      timeout: 5s
      retries: 3

  # --- DVhub container (full-Docker only) ---
  dvhub:
    build: .
    profiles: ["full"]
    ports:
      - "8080:3000"
    environment:
      NODE_ENV: production
      DV_APP_CONFIG: /etc/dvhub/config.json
      DV_DATA_DIR: /var/lib/dvhub
    volumes:
      - ./config:/etc/dvhub
      - dvhub-data:/var/lib/dvhub
    depends_on:
      eos:
        condition: service_healthy
      emhass:
        condition: service_healthy
    restart: unless-stopped

volumes:
  eos-data:
  emhass-data:
  evcc-data:
  dvhub-data:
```

**Activation:**
- Native-only: No Docker needed, just `systemctl start dvhub`
- Hybrid: `docker compose --profile hybrid up -d` + `systemctl start dvhub`
- Full-Docker: `docker compose --profile full up -d`

### Pattern 2: Compose-Manager (child_process)
**What:** Node.js module that controls Docker Compose via CLI
**When to use:** DVhub needs to start/stop/health-check optimizer containers programmatically

```javascript
// Source: Node.js child_process docs + project convention
import { execFile } from 'node:child_process';
import { promisify } from 'node:util';

const execFileAsync = promisify(execFile);

/**
 * @param {string} composePath - Path to docker-compose.yaml
 * @param {string} profile - Active profile name
 */
export function createComposeManager({ composePath, profile = 'hybrid' }) {
  const baseArgs = ['compose', '-f', composePath, '--profile', profile];

  async function run(args) {
    const { stdout, stderr } = await execFileAsync('docker', [...baseArgs, ...args], {
      timeout: 30_000
    });
    return { stdout: stdout.trim(), stderr: stderr.trim() };
  }

  return {
    async up()      { return run(['up', '-d', '--wait']); },
    async down()    { return run(['down']); },
    async ps()      { return run(['ps', '--format', 'json']); },
    async restart(service) { return run(['restart', service]); },
    async isHealthy(service) {
      try {
        const { stdout } = await run(['ps', service, '--format', '{{.Health}}']);
        return stdout.includes('healthy');
      } catch { return false; }
    }
  };
}
```

### Pattern 3: Staggered Optimizer Scheduling
**What:** Prevent concurrent CPU-heavy optimization runs on resource-constrained hardware
**When to use:** Hybrid mode on Pi or low-resource x86

```javascript
// Stagger runs by offsetting each optimizer's schedule
// EOS runs at minute :00, EMHASS at minute :15
// This prevents both hitting CPU limits simultaneously
const STAGGER_OFFSETS_MS = {
  eos: 0,
  emhass: 15 * 60 * 1000  // 15 minutes offset
};
```

### Anti-Patterns to Avoid
- **Using `:latest` tag for optimizer images:** Version drift causes silent breakage; always pin exact versions
- **Running EOS/EMHASS concurrently on Pi:** Both are CPU-intensive Python workloads; stagger scheduling
- **Adding dockerode as dependency:** Violates minimal-dependency principle; CLI wrapper via child_process suffices
- **Hardcoding container URLs in adapters:** URLs already configurable via adapter config; compose-manager should pass URLs from compose service discovery
- **Using `docker-compose` (v1 hyphenated):** Deprecated; use `docker compose` (v2 plugin)

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Container orchestration | Custom container manager | Docker Compose with profiles | Handles networking, volumes, restart, health checks |
| Process supervision | Custom watchdog | systemd (native) / Docker restart policy (container) | Battle-tested, integrated with OS logging |
| Multi-arch image builds | Architecture-specific Dockerfiles | Docker Buildx with multi-platform manifest | Single Dockerfile, automatic arch selection |
| Container health monitoring | Custom health poll loop | Docker Compose healthcheck + compose-manager status | Built-in retry, interval, and health state |
| Service dependency ordering | Custom startup sequencing | Docker Compose depends_on with condition: service_healthy | Declarative, handles retries |

**Key insight:** Docker Compose already solves container lifecycle, health monitoring, resource limits, and dependency ordering. The compose-manager only needs to be a thin CLI wrapper that translates DVhub config into compose commands.

## Common Pitfalls

### Pitfall 1: deploy.resources.limits Ignored Without Docker Compose v2 CLI
**What goes wrong:** Resource limits in compose.yaml silently ignored
**Why it happens:** Old `docker-compose` (v1, Python-based) ignores `deploy` section unless in Swarm mode
**How to avoid:** Require `docker compose` (v2, Go-based). Check with `docker compose version` in install.sh
**Warning signs:** `docker stats` shows no memory limit; optimizer OOM-kills other processes on Pi

### Pitfall 2: EOS Requires seccomp:unconfined and High ulimits
**What goes wrong:** EOS container fails to start or crashes during optimization
**Why it happens:** EOS uses numpy/scipy with OpenBLAS threading that requires elevated limits
**How to avoid:** Include `security_opt: ["seccomp:unconfined"]` and `ulimits: { nproc: 65535, nofile: 65535 }` in compose.yaml
**Warning signs:** Container exits immediately; logs show "resource temporarily unavailable"

### Pitfall 3: npm install vs npm ci for Reproducible Deploys
**What goes wrong:** Different dependency versions on different machines
**Why it happens:** `npm install` resolves ranges; lockfile may drift
**How to avoid:** Use `npm ci --omit=dev` which installs exactly what's in package-lock.json; fail if lockfile is missing or out of sync
**Warning signs:** "peer dependency" warnings; different behavior between dev and production

### Pitfall 4: Docker Volume Permissions on Pi
**What goes wrong:** Container can't write to mounted volumes
**Why it happens:** Docker runs as root inside container, but bind mounts inherit host permissions
**How to avoid:** Use named Docker volumes (not bind mounts) for data; named volumes are managed by Docker with correct permissions
**Warning signs:** Permission denied errors in container logs

### Pitfall 5: Port Conflicts Between Modes
**What goes wrong:** Hybrid mode DVhub can't reach optimizer containers
**Why it happens:** In hybrid mode, DVhub runs on host network, containers on bridge network
**How to avoid:** Map container ports to host ports in compose.yaml; adapters connect via localhost:PORT
**Warning signs:** Connection refused when DVhub tries to reach optimizer

### Pitfall 6: EMHASS ARM64 Solver Issues
**What goes wrong:** EMHASS optimization fails on ARM64
**Why it happens:** Some LP solvers don't have ARM64 builds
**How to avoid:** EMHASS v0.16+ defaults to HiGHS solver which is bundled; ensure v0.16+ is pinned
**Warning signs:** "No solver found" or "GLPK not available" errors

### Compose-Manager Health Check Integration
```javascript
// Compose-manager integrates with optimizer adapter health checks
async function checkOptimizerHealth(composeManager, adapterRegistry) {
  const results = {};
  for (const adapter of adapterRegistry.getAll()) {
    const containerHealthy = await composeManager.isHealthy(adapter.name);
    const adapterHealthy = containerHealthy ? await adapter.healthCheck() : { healthy: false };
    results[adapter.name] = {
      container: containerHealthy,
      api: adapterHealthy.healthy,
      version: adapterHealthy.version
    };
  }
  return results;
}
```

## Open Questions

1. **EOS Exact Latest Stable Tag**
   - What we know: Docker Hub hosts `akkudoktor/eos` with amd64+arm64 support; EOS docs reference version 0.2.0
   - What's unclear: The exact latest stable semver tag (Docker Hub tags page couldn't be scraped)
   - Recommendation: Use `akkudoktor/eos:latest` initially to discover the current tag, then pin it. Add a comment in compose.yaml to check for updates quarterly.

2. **EVCC Config in Container vs DVhub Config**
   - What we know: EVCC v0.300+ supports browser-based config; the EVCC bridge in DVhub reads from EVCC REST API
   - What's unclear: Whether EVCC should be pre-configured via volume-mounted evcc.yaml or left to browser setup
   - Recommendation: Mount a minimal evcc.yaml with network settings; let users complete setup via EVCC browser UI at :7070

3. **EOS Data Volume Layout**
   - What we know: EOS mounts `/data` for persistent storage
   - What's unclear: Exact subdirectory structure and what needs persistence across restarts
   - Recommendation: Use a named Docker volume (`eos-data:/data`); this is safe and Docker manages permissions
